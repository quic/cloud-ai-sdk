

# Network Specialization
The basic idea behind network specialization is to package multiple networks that are compiled with different settings for symbolic variables into the same binary (.qpc) and then select at runtime which network (specialization) to run at inference time. From a systems point of view, this is a very high level optimization: One could imagine that users of the system today could compile multiple networks into separate binaries (.qpcs) and then dynamically select at inference time which binary to load and run. Network specialization eliminates the cost of loading a new binary and replaces it with looking up which entry points to use within the currently loaded binary.

## Use Cases
A primary use case is for NLP networks that have a maximum sequence length parameter that limits the sequence length accepted by the model. In cases where this this maximum sequence length is larger than the input supplied to the network, the input is "padded" to meet the maximum sequence length requirements of the model, and then the majority of the remaining computation and data transfer in the graph operates as-if the input was the full maximum sequence length. For example, we may want to have a system that will accept a maximum sequence length of 256 tokens, but many of the inputs will have a smaller number of tokens, say 16 or fewer. In this scenario, we can package two (or more) networks into the same .qpc:
•	Network compiled for max sequence length = 16
•	Network compiled for max sequence length = 256

Each network in this list will be called a specialization.

Then, at inference time, we can pad inputs to one of the acceptable sizes. The compiled model will then select which network to run, based on the input size. Inputs with sequence length <= 16 can be padded to 16 and then run on the more efficient max sequence length = 16 specialization. Longer input sequences can continue to run on the max sequence length = 256 specialization.

Another CV based use case for network specialization is to send different resolution images through a common classifier. In this case, we may have two (or more) networks packaged in the same .qpc:
•	Resnet compiled for resolution 224x224
•	Resnet compiled for resolution 112x112

At inference time, we can select the appropriate network to run from the input image's resolution.

## Testing network specialization
### Set up
To try these commands locally, a script is provided here:

```python
import json
import os
import numpy as np


def dump_inputs(inputs, input_folder, input_ty):
    os.makedirs(input_folder, exist_ok=True)

    for input_name, possible_dims in inputs.items():
        input_path = os.path.join(input_folder, input_name)
        os.makedirs(input_path, exist_ok=True)

        for dims in possible_dims:
            input_data = np.random.rand(*dims).astype(input_ty)
            dims_str = "_".join([str(d) for d in dims])
            input_data.tofile(os.path.join(input_path, f"{input_name}_{dims_str}.raw"))


def generate_bert_example():
    ################### Save the specialization config file
    specialization_config = {
        "specializations": [
            {"max_seq_len": "128", "batch_size": "1"},
            {"max_seq_len": "256", "batch_size": "2"},
        ]
    }
    with open("bert_specializations.json", "w") as f:
        json.dump(specialization_config, f)

    ################### Save the inputs
    input_ty = np.int64
    inputs = {
        "input_ids": [[1, 128], [2, 256]],
        "input_mask": [[1, 128], [2, 256]],
        "segment_ids": [[1, 128], [2, 256]],
    }

    dump_inputs(inputs, "inputs/bert", input_ty)

    ################### Save the json_io_shapes file
    bert_io_shapes = {
        "IO-files": [
            [
                {
                    "path": "inputs/bert/input_ids/input_ids_1_128.raw",
                    "dims": [1, 128],
                    "elem-size": 8,
                },
                {
                    "path": "inputs/bert/input_mask/input_mask_1_128.raw",
                    "dims": [1, 128],
                    "elem-size": 8,
                },
                {
                    "path": "inputs/bert/segment_ids/segment_ids_1_128.raw",
                    "dims": [1, 128],
                    "elem-size": 8,
                },
                {
                    "path": "out_cat_1_128_dyn.bin",
                    "dims": [1, 2],
                    "data-type": "float",
                    "io-direction": "out",
                },
            ],
            [
                {
                    "path": "inputs/bert/input_ids/input_ids_2_256.raw",
                    "dims": [2, 256],
                    "elem-size": 8,
                },
                {
                    "path": "inputs/bert/input_mask/input_mask_2_256.raw",
                    "dims": [2, 256],
                    "elem-size": 8,
                },
                {
                    "path": "inputs/bert/segment_ids/segment_ids_2_256.raw",
                    "dims": [2, 256],
                    "elem-size": 8,
                },
                {
                    "path": "out_cat_2_256_dyn.bin",
                    "dims": [2, 2],
                    "data-type": "float",
                    "io-direction": "out",
                },
            ],
        ]
    }

    with open("bert_io_shapes.json", "w") as f:
        json.dump(bert_io_shapes, f)


def generate_resnet_example():
    specialization_config = {
        "specializations": [
            {"batch": "1", "height": "112", "width": "112"},
            {"batch": "1", "height": "224", "width": "224"},
        ]
    }

    with open("resnet_specializations.json", "w") as f:
        json.dump(specialization_config, f)

    ################### Save the inputs
    input_ty = np.float32
    inputs = {
        "img": [[1, 3, 112, 112], [1, 3, 224, 224]],
    }

    dump_inputs(inputs, "inputs/resnet", input_ty)

    ################### Save the json_io_shapes file
    resnet_io_shapes = {
        "IO-files": [
            [
                {
                    "path": "inputs/resnet/img/img_1_3_112_112.raw",
                    "dims": [1, 3, 112, 112],
                    "elem-size": 4,
                },
                {
                    "path": "out_112.bin",
                    "dims": [1, 1000],
                    "data-type": "float",
                    "io-direction": "out",
                },
            ],
            [
                {
                    "path": "inputs/resnet/img/img_1_3_224_224.raw",
                    "dims": [1, 3, 224, 224],
                    "elem-size": 4,
                },
                {
                    "path": "out.bin",
                    "dims": [1, 1000],
                    "data-type": "float",
                    "io-direction": "out",
                },
            ],
        ]
    }

    with open("resnet_io_shapes.json", "w") as f:
        json.dump(resnet_io_shapes, f)


def main():
    generate_bert_example()

    generate_resnet_example()


if __name__ == "__main__":
    main()
```

### ResNet specialization
Logically, running inferences with network specialization should give the same results as running single inferences against the a single, unspecialized network.

For example, we can compile a resnet model with variable batch, height, width parameters for network specialization as follows:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
    -network-specialization-config=resnet_specializations.json \
    -json-input-file=resnet_io_shapes.json \
    -convert-to-fp16
```

By default, this will compile and run the network with the inputs specified in the resnet_io_shapes.json file on the native execution path.

In this basic example, the file resnet_specializations.json could contain:
```json
{
    "specializations": [
        {
            "batch": "1",
            "height":"224",
            "width":"224"
        },
        {
            "batch": "1",
            "height":"112",
            "width":"112"
        }
    ]
}
```

This will create a single .qpc with 2 logical networks embedded inside: A 112x112 resnet network and a 224x224 resnet network. To run inferences against this network, one would need to prepare 2 sets of inputs: 112x112 inputs and 224x224 inputs. Then, we can run each input against the single .qpc. For the example above, we create 2 input .json files: inputs-112.json and inputs-224.json, each pointing to prepared images with resolutions 112x112 and 224x224. The format for these input files is:

```json
{"IO-files": [
    [
       { "path":"inputs/resnet/img_1_3_112_112.raw", "dims":[1,3,112,112], "elem-size":4},
       { "path":"out_112.bin",  "dims":[1,1000], "data-type":"float", "io-direction":"out"}
    ],
    [
        { "path":"inputs/resnet/img_1_3_224_224.raw", "dims":[1,3,224,224], "elem-size":4},
        { "path":"out_224.bin", "dims":[1,1000], "data-type":"float", "io-direction":"out"}
    ]  
]}
```
The results from running these networks should be the same as-if we compiled 2 separate networks with (batch,height,width) equal to (1,224,224) and (1,112,112), and feeding appropriately sized inputs.

#### Running on hardware
The above example will default to running on the native path because we have not told QAicExec to use the hardware path.
If the machine you are using has a Qranium card available, you can simply add the flag "-aic-hw" to the QAicExec command:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
    -network-specialization-config=resnet_specializations.json \
    -json-input-file=resnet_io_shapes.json \
    -convert-to-fp16 \
    -aic-hw
```
### BERT specialization
In addition to image resolution, another typical use case for network specialization is sequence length in language models.

For the example of BERT, the specialization configuration describes two networks with different batch sizes and sequence lengths:
```json
{
    "specializations": [
        {
            "max_seq_len": "128",
            "batch_size": "1"
        },
        {
            "max_seq_len": "256",
            "batch_size": "2"
        }
    ]
}
```
BERT also has multiple inputs, so the json-io-shapes file looks slightly different:
```json
{
    "IO-files": [
        [
            { "path": "inputs/bert/input_ids/input_ids_1_128.raw", "dims": [1,128], "elem-size": 8 },
            { "path": "inputs/bert/input_mask/input_mask_1_128.raw", "dims": [1,128], "elem-size": 8 },
            { "path": "inputs/bert/segment_ids/segment_ids_1_128.raw", "dims": [1, 128], "elem-size": 8},
            { "path": "out_cat_1_128_dyn.bin", "dims": [1,2],"data-type": "float","io-direction": "out"}
        ],
        [
            { "path": "inputs/bert/input_ids/input_ids_2_256.raw", "dims": [2,256],"elem-size": 8},
            { "path": "inputs/bert/input_mask/input_mask_2_256.raw", "dims": [2,256],"elem-size": 8},
            { "path": "inputs/bert/segment_ids/segment_ids_2_256.raw", "dims": [2,256],"elem-size": 8},
            { "path": "out_cat_2_256_dyn.bin", "dims": [2,2], "data-type": "float", "io-direction": "out"
            }
        ]
    ]
}
```
The compile command looks the same:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
    -network-specialization-config=bert_specializations.json \
    -json-input-file=bert_io_shapes.json
```

## Quantization
In order to generate a quantization profile when using multiple specializations, the user needs to run QAicExec twice: once to dump the quantization file and a second time to use the quantization file.

For example for the BERT model we are using in the above example, the user can dump the quantization profile with the following command:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
    -network-specialization-config=bert_specializations.json \
    -json-input-file=bert_io_shapes.json \
    -dump-profile=bert_pgq.yaml
```

And then to use the profile, the -dump-profile option needs to be replaced by -load-profile:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
    -network-specialization-config=bert_specializations.json \
    -json-input-file=bert_io_shapes.json \
    -load-profile=bert_pgq.yaml
```
Note that the IO shapes file must include at least one set of inputs per specialization.
Per-specialization compile options

QAicExec also provides support for per-specialization compile options. For example, for the ResNet above we could specify in the specializations configuration file:
```json
{
    "specializations": [
        {
               "batch": "1",
            "height":"112",
            "width":"112",
            "compile_opts": "-ols=2"
        },
        {
            "batch": "1",
             "height":"224",
            "width":"224",
            "compile_opts": "-ols=4"
        }
    ]
}
```

To verify that the compiler option is propagated correctly we can ask qaic-exec to store the graph IR by exporting the following environment variable:
```bash
export QAIC_COMPILER_OPTS_UNSUPPORTED="-debug-glow-only=aic-dump-final-ir -aic-dump-llvm-ir-postopt -aic-dump-graphs-dir=graphs/"
```

Before running qaic-exec and checking that the intended optimization has been enabled correctly:
```bash
/opt/qti-aic/exec/qaic-exec \
    -m=/path/to/model.onnx  \
        -network-specialization-config=resnet_specializations.json \
        -json-input-file=resnet_io_shapes.json \
        -compile-only \
        -aic-hw
```
Currently -ols is the only option that is supported this way.

### Running Inferences
It is the user's / application's responsibility to preprocess the inputs into appropriate shapes that can be accepted by one of the specialized networks.

For example, if we have a CV network that accepts 112x112 and 224x224 specializations, if we would like to run an inference on a 176x176 image, it is the application's responsibility to either clip / downscale to a 112x112 image, or pad / upsample to a 224x224 image. 

This should be thought of in the same way as if we have a single 224x224 network available, it is the applications responsibility to preprocess the image to match this size exactly.
